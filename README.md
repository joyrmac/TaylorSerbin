## Key Concepts

### The Doctrine
Five non-negotiable principles that govern all work:

1. **Clarity beats cleverness** — Say the thing directly
2. **Restraint beats excess** — Show discipline, not volume
3. **Finished beats polished** — Real over perfect
4. **Authority is quiet** — No hype, no self-congratulation
5. **SEO serves structure** — Search optimization supports user experience, not the reverse

See `docs/OPERATING_MANUAL.md` for the full philosophy.

### The Brand System
Two complementary brands with one unified authority:

- **Personal Brand (Jason):** Warm, authentic, peer-to-peer. Builds trust in motorcycle and lifestyle communities.
- **Corporate Brand (Firm):** Professional, serious, proven. Converts research into clients.

They work together: Personal brand builds relationship → Corporate brand handles legal conversion. See `BRANDING_GUIDE.md` for complete visual and verbal standards.

### The Content Model
**Raw Feed System:** Jason provides raw content (photos, videos, context). The agency handles execution (editing, caption writing, scheduling, engagement).

This keeps Jason focused on living the brand. The agency focuses on systematic execution.

See `strategy/personal-brand/CONTENT_STRATEGY.md` for the complete model.

### Content Pillars (Instagram & Social)
Jason's content is distributed across four pillars using a **40-20-20-20 model**:

- **Rider's Life (40%):** Motorcycle community, rides, events, authenticity
- **Gourmet Litigator (20%):** Cooking, recipes, entertaining
- **Cigars & Five-Stand (20%):** Cigar collecting, clay shooting, lifestyle
- **Alligator Wisdom (20%):** Legal tips, motorcycle safety, firm value connection

This balance shows the full person while maintaining focus on the primary audience. See `strategy/personal-brand/CONTENT_STRATEGY.md`.

### Platform Cadence
Different platforms have different rhythms:

- **Instagram:** 4–5 posts/week (primary platform, main stage)
- **LinkedIn:** 2–3 posts/week (professional credibility, authority)
- **Facebook:** 3–4 posts/week (community + past clients)

See `strategy/personal-brand/PLATFORMS.md` for per-platform detail.

## How Decisions Work

**All decisions trace back to the doctrine.** Here's the hierarchy:

1. **Operating Manual** (North Star, philosophy, principles)
2. **Execution Standards** (tone, length, publishing rules)
3. **Brand Authority & Guidelines** (voice, authority protection)
4. **Strategy Documents** (specific initiatives)
5. **Team decisions** (day-to-day execution)

**When deciding something:**
- Start at level 1 (does the Operating Manual govern this?)
- If not, check level 2 (does Execution Standards address it?)
- Continue down until you find the rule
- If no rule exists, propose one and document in git

See `docs/OPERATING_MANUAL.md` for the complete decision framework.

## Working Norms

### Decisions
- Cite the manual when making calls
- When in doubt, default to "clarity beats cleverness"
- Document decisions as git commits (not emails or Slack)

### Changes
- Only modify doctrine when a pattern repeats (not for one-off preferences)
- Propose changes to the repo, don't make assumptions
- Get agreement before changing non-negotiables

### Publishing
- Work is done when it passes the doctrine check, not when it feels done
- Always run through `CONTENT_REVIEW_CHECKLIST.md`
- Every post has metadata (pillar, platform, date, performance tracking)

### Feedback
- Push back on anything that violates the North Star
- Feedback should be constructive and reference the manual
- Disagreement is encouraged if it serves clarity

## FAQ

**Q: Can I modify the Operating Manual?**  
A: No. It's the North Star. Propose changes only if a clear pattern repeats, and document in git.

**Q: What if I find a rule that contradicts the doctrine?**  
A: Escalate to the team lead. The doctrine governs all work—contradiction is a problem.

**Q: How do I create a new file or section?**  
A: Reference this repo structure and file naming conventions. Keep it organized. Document purpose at the top of new files.

**Q: When should I push to git?**  
A: Every time you create a new strategy, finish a content piece, or modify doctrine. Use clear commit messages.

**Q: What's the difference between WHAT_WE_KNOW and WHAT_WE_NEED?**  
A: WHAT_WE_KNOW is facts about Jason, the firm, community. WHAT_WE_NEED is prioritized information gaps we're still gathering.

**Q: Who approves content?**  
A: See ROLES.md for the content review process. Typically: Social Manager → Reviewer (approves tone) → Publish.

**Q: How often should the strategy change?**  
A: Quarterly reviews. Real-time adjustments only for documented issues. Evolution is planned, not reactive.
